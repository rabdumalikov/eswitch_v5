9 **Eswitch** it is header-only library which was written on c++20. This code it is my attempt to _enhance_ **switch statement** capability with similar syntax.

33 \section tutorial-quick-start Quick Start
34 
35 --------------------------------------------------------------------------------
36 **eswitch_v5** is a library which emulate _C++_ functionality i.e. **switch statement** and through this emulation I was able to overcome **native switch** limitations:
37 
38 - one parameter per **native switch**
39 - the parameter restricted to only _integral_ types( **int**, **char**, **enum** ... ).
40 
41 **eswitch_v5** supports any number of _parameters_ and almost without restriction on their _type_
42  as long as the _type_ is **comparable**( i.e. has **operator==**, **operator!=** and so on ).
43  Additionally, my library allow to match for **std::any, std::variant<...>, polymorphic type match** and
44  **std::regex**. Also it allows to withdraw values from those types, of course if match was successful.
45 
46 ### Motivation
47 
48 I don't see any good reason why **switch statement** in **C++** is so limited, whereas other
49 statements such as **if, while, for** have no such limitations and they allow to compose and
50 test complex condition. Moreover in terms of _**assembler**_ output, **if** and **switch**
51 statements give the same output, thus the performance also the same. If there isn't any
52 benefits to keep **switch statement** limited, so why the things the way they are? I also
53 don't think that **switch statement** is limited because of compatability with **C**, because
54 syntax of **if statement** is still compatible with **C** even though it was extended in **C++17**, this extention allows to declare variable
55 within **if statement** like this:
56 ```cpp
57 if( std::smatch mt; std::regex( text, mt, rgx ) ) {...}
58 ```
59 On top of that the internet is full of questions:
60 - [Why strings cannot be used in **switch statement**](https://stackoverflow.com/questions/650162/why-the-switch-statement-cannot-be-applied-on-strings),
61 - [How to compose complex condition in **switch statement**](https://stackoverflow.com/questions/68578/multiple-cases-in-switch-statement)
62 - etc.
63 
64 And there was even the [proposal](http://www.open-std.org/jtc1/sc22/wg21/docs/papers/2013/n3627.html) for **C++ standard committee** to allow to use _complex types_( like **string**, **complex numbers**, etc ) within **switch statement** and **committee** <span style="color:green"> *agreed* </span> about **importance of this topic**.
65 But somehow the author stopped his work toward this direction.
66 
67 
68 There is more, compilers such as **clang** and **gcc** have non-standard extention for matching [**case ranges**](https://gcc.gnu.org/onlinedocs/gcc/Case-Ranges.html).
69 ```cpp
70 switch( num ) switch( ch )
71 { {
72  case 1 ... 5: break; case 'A' ... 'Z': break;
73  case 10 ... 99: break; case 'a' ... 'z': break;
74 }; };
75 ```
76 The evidence above tell us that people don't like limitations of **switch statement**, and they're trying to overcome
77 them with different approaches( by implementing **non-standard extension** or writing **proposals** or just **wondering** ).
78 
79 For modern language like **swift** it is not an issue, because they have pretty advanced **switch statement**.
80 
81 Based on those factors I decided to write my own implementation of **enhanced switch** or just **eswitch**.
82 In my implementation I tried to aim all the **limitations** and leave the syntax of **eswitch** as close as
83 possible to _native C++_ **switch statement**. And last but not least, another my priority was the performance
84 of **eswitch**, which shouldn't differ by much from **native switch**.
85 
86 ### Conventions used in this document:
87 
88 In all code examples, I omit the namespace prefixes for names in the **eswitch_v5** and **std** namespaces. Also I'm going to use following name conventions:
89 - **native switch** => _native C++_ **switch statement**
90 - **eswitch** => my custom implementation of **enhanced switch statement**
91 
92 ### Similarities
93 
94 As I have mentioned that I've tried within my implementation to resemble _C++_ **switch statement** syntax. And I think I was able to achieve this, except the places where I was either limited by language
95 or intentionally tried to avoid certain behavior of **switch statement** in _C++_ like default **fallthrough**. Compare:
96 
97 ```cpp
98 switch( num )
99 {
100  case 1: {...} break;
101  case 2: {...} break;
102  default: {...} break;
103 };
104 ```
105 
106 with
107 
108 ```cpp
109 eswitch( num )
110 (
111  Case( 1 ) {...},
112  Case( 2 ) {...},
113  Default {...}
114 );
115 ```
116 **Note That:** _Native_ **switch statement** in _C++_ has _explicit_ **break** and _implicit_ **fallthrough**
117 which proven to be error-prone approach. Since developers sometimes forget to use
118 **break** and because of this their code doesn't work the way it was intended.
119 Thus in my implementation I reversed this concept i.e. **eswitch** has _implicit_ **break**
120 and _explicit_ **fallthrough**. Compare:
121 
122 ```cpp
123 switch( num )
124 {
125  case 1: {...}
126  case 2: {...} break;
127  default: {...} break;
128 };
129 ```
130 
131 with
132 
133 ```cpp
134 eswitch( num )
135 (
136  Case( 1 ){...} ^ fallthrough_,
137  Case( 2 ){...},
138  Default {...}
139 );
140 ```

142 ### Keywords
143 
144 | Name | Description |
145 | :---: | :---: |
146 | ***eswitch*** | _accepts list of input parameters_ |
147 | ***Case*** | _accepts **condition** to check and body next to it will be executed if **condition** matched_ |
148 | ***Default*** | _body next to it will be executed if nothing else matched_ |
149 | ***fallthrough_*** | _next body will be executed without checking its **condition**_ |
150 | ***any_from*** | _accepts values to choose from_ |
151 | ***is<sometype>*** | _used within **Case** for matching types like **std::any**, **std::variant<...>** and **polymorphic type match**_ |
152 | ***_r*** | _user defined literal for **std::regex**_ |
153

155 
156 ### Syntax
157 
158 ``` cpp
159 eswitch( __params__ )
160 (
161  Case( __conditions__ )( __value__ ) { ... } ^ __options__,
162  Default { ... }
163 );
164 ```
165 
166 Where
167 
168 | Name | Details |
169 | :---: | :---: |
170 | <span style="color:blue">__params__</span> | list of parameters( i.e. param1, param2, ..., param_n ) |
171 | <span style="color:blue">__conditions__</span> | match **in order**( i.e. \_1 == smth1 \|\| \_2 == smth2 \|\| ... )<br>or match via **any_from**( i.e. \_1 == **any_from**( smth1, smth1, ... )<br>or match via **predicate**( i.e. ( pred1, \_1 ) && ( pred2, \_2 ) && ... ) |
172 | _**Optional:**_<br><span style="color:blue">__value__</span> | withdrawn values from: **std::any**, **std::variant<...>**,<br> **polymorphic match** or **std::regex match**. |
173 | _**Optional:**_<br><span style="color:blue">__options__</span> | [**left empty** => _break_] or <br> [**fallthrough_** => _execute body of the following case_] or <br> [**likely_** which will be introduced in future] |
174 
175 ### Features
176 
177 - **implicit break** - you don't have to specify it
178 
179 - **fallthrough** - is *explicit*
180 
181 - **composing condition** with: **&&** , **\|\|** , **==** , **!=** , **>** , **<** , **<=** , **>=**
182 - **param matching via indexing and without**
183 - **comparison made easy**
184 - **match for**: _std::any, std::variant, std::regex_
185 - **match and withdraw value from**: _std::any, std::variant, std::regex_
186 
187 #### Params referencing via indexes
188 
189 ```cpp
190 eswitch( p1, p2 )
191 (
192  Case( _1 == true && _2 == true ) {...},
193  Case( _2 != true || _1 == true ) {...}
194 );
195 // i.e.
196  if( p1 == true && p2 == true ) {...}
197 else if( p1 == true || p2 == true ) {...}
198 ```
199 
200 
201 #### If indexes aren't provided, then it is assummed that they're sequential
202 
203 ```cpp
204 eswitch( p1, p2 )
205 (
206  Case( true, false ) {...}, // same as "Case( _1 == true && _2 == false )"
207  Case( false ){...}
208 );
209 ```
210 
211 #### Matching via predicates:(it could be either a lambda or a free function)
212 
213 ```cpp
214 eswitch( p1, p2 )
215 (
216  Case( ( is_odd, _1 ) && ( is_odd, _2 ) ) {...},
217  Case( ( is_negative( _1, _2 ) ){...}
218 );
219 
220 // i.e.
221  if( is_odd( p1 ) && is_odd( p2) ) {...}
222 else if( is_negative( p1, p2 ) ) {...}
223 ```
224 
225 #### Check in range via predicate
226 
227 ```cpp
228 eswitch( p1 )
229 (
230  Case( _1.in( 1, 10 ) ) {...},
231  Case( _1.within( 11, 20 ) ) {...}
232 );
233 // OR
234 eswitch( p1 )
235 (
236  Case( _1 > 1 && _1 < 10 ) {...},
237  Case( _1 >= 11 && _1 <= 20 ) ) {...}
238 );
239 
240 // i.e.
241  if( p1 > 1 && p1 < 10 ) {...}
242 else if( p1 >= 11 && p1 <= 20 ) {...}
243 ```
244 
245 #### Any_from helper function
246 
247 ```cpp
248 eswitch( file_extention )
249 (
250  Case( any_from( "cpp", "cc", "c++", "cxx", "C" ) ) {...},
251  Case( any_from( "h", "hpp", "hh", "h++", "hxx", "H" ) ) {...},
252  Default {...}
253 );
254 ```
255 
256 #### Default case
257 
258 ```cpp
259 eswitch( p1 )
260 (
261  Case( false ) {...},
262  Default {...}
263 );
264 ```

268 ```cpp
269 bool val = eswitch( p1 )

275 #### Match for std::any or std::variant
276 
277 ```cpp
278 eswitch( any_ )
279 (
280  Case( is< int >{} ) {...}, // will be executed in 'std::any' contain 'int'
281  Case( is< string >{} ) {...} // will be executed in 'std::any' contain 'std::string'
282 );
283 ```
284 
285 #### Match and withdraw value from std::any or std::variant
286 
287 ```cpp
288 eswitch( any )
289 (
290  Case( is< int >{} )( const int value ) {...}, // Note that keyword 'auto' isn't allowed( i.e. code won't compile )
291  Case( is< string >{} )( const string & value ) {...}
292 );
293 ```
294 
295 #### Match for polymorphic types
296 
297 ```cpp
298 eswitch( base_ptr )
299 (
300  Case( is< circle >{} )( circle * c ) { c->draw(); },
301  Case( is< square >{} )( square * s ) { s->draw(); }
302 );
303 // OR
304 eswitch( base_ref )
305 (
306  Case( is< circle >{} )( circle & c ) { c.draw(); },
307  Case( is< square >{} )( square & s ) { s.draw(); }
308 );
309 ```
310 
311 #### Match for std::regex
312 
313 ```cpp
314 eswitch( text )
315 (
316  Case( R"(\w*)"_r ) { return "message"; },
317  Case( R"(\d*)"_r ) { return "number"; }
318 );
319 ```
320 
321 #### Match and withdraw values from std::regex
322 
323 ```cpp
324 eswitch( text )
325 (
326  /// NOTE that "vector<string>"" and "std::smatch" contain same values
327  Case( R"((\w*))"_r )( vector< string > && match ){ return match[1]; }
328  Case( R"((\d*))"_r )( vector< string > && match ){ return match[1]; }
329 );
330 ```
